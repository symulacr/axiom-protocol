## Trace 1: IntelligentData

The core ERC-7857 on-chain metadata unit: a `(dataDescription, dataHash)` pair that links an iNFT's encrypted blob on 0G Storage to its on-chain record.

### 1.1 Creation

**User-side:** The dataDescription and encrypted blob are produced by the application frontend or CLI, then a root hash (dataHash) is computed from the blob on 0G Storage. The oracle registers the hash via:

```
POST /v1/agents/mint  —  apps/oracle/src/server.ts:233-241
```
Body: `{ dataHash: "0x..." }`
Validation: `mintDataHashSchema` Zod schema (`apps/oracle/src/route-schemas.ts:25-27`)
- `dataHash` must be a 32-byte hex string via `hexViem` + length check at line 235

The oracle marks it seen: `storage.markDataHashSeen(dataHash)` (line 239).

**On-chain mint:** `AxiomAgentNFT.mint(iDatas, to)` — `apps/contracts/src/AxiomAgentNFT.sol:189-200`

- Calls `_updateData(tokenId, iDatas)` (line 198) → stores in ERC-7201 slot
- Alternative: `mintWithRole()` gated by `MINTER_ROLE` (lines 202-220)
- Assignes token ID via `_incrementTokenId()` (line 194)

## Trace 2: OwnershipProof / AccessProof (EIP-712 Typed Data)

- `apps/oracle/src/signer.ts:71-74` — `TeeSigner.signOwnership(input)`: computes EIP-712 digest, signs with secp256k1 private key via ethers `Wallet.signingKey.sign(digest)`
- Called from two HTTP endpoints:
  - `/v1/transfer-validity` — `apps/oracle/src/server.ts:106-114`: used during transfer flow, includes blob re-encryption
  - `/v1/ownership` — `apps/oracle/src/server.ts:217-225`: standalone ownership proof signing

**AccessProof** — signed by the receiver (browser wallet via `signTypedData_v4` or backend):

## Trace 5: Encrypted Agent Blob

The full lifecycle of an iNFT's encrypted metadata: encryption → 0G Storage → re-encrypted transfer.

### 5.1 Creation (Encryption)

**AES-256-GCM encryption** — `apps/oracle/src/crypto/aes-gcm.ts:17-28`:
```typescript
aesGcmEncrypt(key: Uint8Array, plaintext: Uint8Array): EncryptedPayload
```
- Key requirement: exactly 32 bytes (line 18 validation)
- Random 12-byte IV (`randomBytes(IV_LENGTH)`, line 19)
- Output layout: `iv (12) || ciphertext || authTag (16)`
- Concatenation helper: `concatEncrypted(payload)` — `aes-gcm.ts:42-44`

**Upload to 0G Storage:** The encrypted blob is uploaded via:
```typescript
storage.upload(newBlob) // returns { rootHash: string }
```
— `apps/oracle/src/server.ts:98` (transfer-validity endpoint)

The `rootHash` becomes the `dataHash` on-chain.

### 5.2 Transfer Flow (Re-encryption)

The oracle's `/v1/transfer-validity` endpoint orchestrates the complete re-encryption cycle:

**apps/oracle/src/server.ts:60-129**:

1. **Download old blob** from 0G Storage: `storage.download(oldDataUri)` (line 85)
2. **Parse encrypted format:** `parseEncrypted(oldBlob)` — `crypto/aes-gcm.ts:46-52`: splits blob into `{ iv, ciphertext, authTag }`
3. **Decrypt with old key:** `aesGcmDecrypt(oldDataKey, oldEnc)` — `crypto/aes-gcm.ts:30-40`:
   - Sets auth tag, creates decipher, returns plaintext `Uint8Array`
4. **Generate new key:** `crypto.randomBytes(32)` (line 95)
5. **Re-encrypt with new key:** `aesGcmEncrypt(newDataKey, oldPlaintext)` (line 96)
6. **Concatenate:** `concatEncrypted(newEnc)` → single `Uint8Array` blob (line 97)
7. **Upload new blob** to 0G Storage: `storage.upload(newBlob)` (line 98)
8. **Mark seen:** `storage.markDataHashSeen(newDataHash)` (line 100)
9. **ECIES seal:** `sealKeyForReceiver(targetPubkeyBytes, newDataKey)` (line 103)

### 5.3 ECIES Key Wrapping

**sealKeyForReceiver** — `apps/oracle/src/crypto/ecies.ts:25-27`:
- Takes 64-byte uncompressed receiver public key + 32-byte data encryption key
- Converts to compressed form (33 bytes): `toCompressed()` (line 14-22) using `noble/secp256k1.ProjectivePoint`
- Calls `eciesjs.encrypt(receiverPubkeyCompressed, dataEncryptionKey)` (line 26)

**unsealKeyForReceiver** — `apps/oracle/src/crypto/ecies.ts:29-31`:
- `eciesjs.decrypt(receiverPrivateKey, sealedKey)` recovers the data encryption key

### 5.4 Transfer Verification (On-chain)

The sealed key is delivered on-chain via the OwnershipProof flow:

1. `sealedKey` becomes part of `OwnershipProofInput.sealedKey`
2. Signed by TEE → verified in `AxiomTeeVerifier.verifyTransferValidity()` — `AxiomTeeVerifier.sol:215-230`
3. Output struct carries `sealedKey` — `AxiomTeeVerifier.sol:269`
4. `_proofCheck` collects sealed keys — `ERC7857Upgradeable.sol:124`
5. `_transfer` emits `PublishedSealedKey(to, tokenId, sealedKeys)` — `ERC7857Upgradeable.sol:131`
6. The new owner receives the event with the sealed key and can:
   - Use their private key to `unsealKeyForReceiver()` and recover the data encryption key
   - Download the blob from 0G Storage using `newDataHash`
   - `aesGcmDecrypt()` to recover the plaintext

### 5.5 Validation

| Stage | Location | Check |
|-------|----------|-------|
| AES key length | `crypto/aes-gcm.ts:18,31` | `key.length === 32` |
| IV + auth tag length | `crypto/aes-gcm.ts:47` | `blob.length < 28` reject |
| Blob parse | `crypto/aes-gcm.ts:46-51` | Splits at byte offsets |
| ECIES pubkey format | `crypto/ecies.ts:15-22` | 64, 33, or 65 bytes → secp256k1 point validation |

### 5.6 Persistence

- **Blob storage:** 0G Storage (content-addressable, root hash as identifier)
- **Key chain:** Never stored on-chain (ephemeral in oracle memory during transfer)
- **Sealed key:** Passed through on-chain `PublishedSealedKey` event (ephemeral, in event log)
- **Addressing:** `dataHash` (0G Storage root hash) stored on-chain in `IntelligentData.dataHash`

### 5.7 Serialization

- **Encrypted blob:** Binary `Uint8Array` = `iv (12) || ciphertext || authTag (16)` — custom format (`apps/oracle/src/crypto/aes-gcm.ts:42-44`)
- **ECIES seal:** Binary ciphertext with ephemeral pubkey prefix (eciesjs format)
- **On-chain:** `bytes` for sealedKey, `bytes32` for dataHash
- **HTTP:** Hex-encoded (`0x...`) for signatures, hex for URIs, base64 for encryption keys

### 5.8 Cross-Boundary Transport

- **Oracle ↔ 0G Storage:** HTTP (indexer RPC for download/upload)
- **Oracle → Chain:** The `dataHash` goes on-chain via `AxiomAgentNFT.mint()` or `update()` — EVM transaction
- **Oracle → Transfer Initiator:** HTTP response with `{ newDataUri, newDataHash, sealedKey, ownershipSignature, ... }`
- **Contract → Event Log:** `PublishedSealedKey` event (EVM log)
- **Receiver:** Reads event log → downloads blob from 0G Storage → unseals key via ECIES → decrypts with AES-256-GCM
