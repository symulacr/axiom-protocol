## Why a plain-CREATE redeploy (not CREATE2)?

The original AxiomTeeVerifier at `0xE0D0F3…` is the pre-fix bytecode. A
CREATE2 deployment with the original salt would deterministically map to that
same address, but that address already has runtime code on-chain; per
[EIP-684](https://eips.ethereum.org/EIPS/eip-684) a second `CREATE2` to a
non-empty address MUST revert. We could:

1. **Brute-force 2^160 salts to find one that hashes to a fresh address that
   also happens to be empty on-chain** — infeasible.
2. **Use plain `new AxiomTeeVerifier(...)`** — a fresh CREATE, address is the
   first available from the operator's next nonce. This is what the task
   recommends and what we did. The new address is logged and the proxy owner
   must follow up with a `setAxiomTeeVerifier(newAddr)` admin tx (see "Open
   follow-ups" below).

The new AxiomTeeVerifier address (`0xb8015C0…`) was predicted from
operator nonce 202 + 1 = 203 (i.e. the address corresponding to nonce 203 of
the operator). The script logs the pre-broadcast nonce so the address is
recoverable from `cast compute-address <operator> --nonce 202` for
verification (see "Verification" below).

## What this fixes (BUGS.md additions)

See `apps/contracts/test/BUGS.md` for the formal entries:

- **BUG-TEE-FIX-03** — BUG-TEE-13D-01 + BUG-TEE-13D-02 are now live-fixed
  (the redeploy brought the new bytecode on-chain). The new verifier
  enforces the `_checkValidUntil` EIP-712 deadline gate and reverts on
  expired / too-far-future proofs.
- **BUG-PAY-FIX-02** — BUG-PAY-13C-01 is now live-fixed
  (the PaymentProcessor is deployed at `0x4357391BB192Fc4A2400cd2cD73aa08d4c9d08d8`
  with non-zero code; the pre-recorded `0xEf1bA8…` is still empty on-chain
  but is no longer the canonical address — the .env points at the new one).
- **BUG-PAY-NEW-01** — PaymentProcessor needs a paymentToken, but
  AxiomPaymentProcessor's constructor reverts on `paymentTokenAddr == 0` and
  0G Galileo has no bridged USDC.e / USDG yet. We deploy a test-only
  `AxiomMockUSDC` mintable ERC-20 alongside the processor. On 0G Aristotle
  mainnet (chainId 16661) this must be replaced with the real USDC.e / USDG.
  The MockUSDC is owner-mintable by anyone (testnet-only — no value).

## New findings

- **AxiomAgentNFT proxy still references the pre-fix verifier** (this is the
  expected Wave 11 / Wave 12 behavior, but it's a CLASS of bug we hadn't
  explicitly captured). The proxy owner must call `setAxiomTeeVerifier(newAddr)`
  to wire the new verifier in. See "Open follow-ups" below.
  Classification: (b) LACK-OF-FOLLOWUP — a redeploy is a 2-step operation
  (deploy + wire) and the wiring step requires a separate admin tx.
- **Plain-CREATE predict reproducibility** — the script logs the operator's
  pre-broadcast nonce (202) so the predicted address is recoverable from
  `cast compute-address <operator> 202`. The actual on-chain address
  matched the prediction exactly. This is a clean win for `forge script` and
  is a better reproducibility story than relying on CREATE2 salt brute-force.
- **Multi-contract script requires `--tc ContractName`** — when a script file
  has more than one contract (here `AxiomMockUSDC` + `DeployPaymentProcessor`),
  `forge script` errors with "Multiple contracts in the target path. Please
  specify the contract name with `--tc ContractName`." Discovered while
  broadcasting the PaymentProcessor; fixed by adding `--tc DeployPaymentProcessor`.
  Adding `BUG-INFRA-16A-01` to BUGS.md as a Foundry quirk (not a chain bug).

1. **Wire the new verifier into the AxiomAgentNFT proxy** (the proxy
   currently stores `verifier = 0xE0D0F3…` and is owner-gated).
   The owner is `0x4373…` (the operator). The exact call is documented in
   `~/og/.env` at the bottom:
   ```bash
   cast send $AGENT_NFT_ADDRESS "setAxiomTeeVerifier(address)" $AXIOM_TEE_VERIFIER \
     --rpc-url $OG_RPC_URL --private-key $DEPLOYER_PK --legacy --gas-price 3000000000
   ```

   to the new (post-fix) verifier. Until this is run, the proxy still

   E2E flow. **This MUST be done before the Wave 16 E2E can validate the
   validUntil behavior end-to-end.**

   payer to have an ERC-20 `approve(payForAgent, amount)` then a
   `payForAgent(tokenId, payer, amount)` call. For the E2E, mint some
   `axmUSDC` to the test wallets:

   ```bash
     <test_wallet> 10000000000000000000000 \
     --rpc-url $OG_RPC_URL --private-key $DEPLOYER_PK --legacy --gas-price 3000000000
   ```
